# Keeper Privacy Improvements

현재 구조는 게임의 재미와 UX를 유지하면서도 비교적 적은 상호작용으로 진행할 수 있다는 장점이 있다.
다만 keeper 서버가 비공개 정보를 잠시 보관한다는 점에서 프라이버시와 신뢰 리스크가 남는다.

아래 개선안들은 게임 플로우를 크게 바꾸지 않으면서, 프라이버시를 더 강화할 수 있는 현실적인 방법들이다.

## 1. 저장 최소화

keeper 서버가 `choices`, `order`, `salt` 같은 원본 데이터를 오래 보관하지 않도록 한다.

- reveal 또는 eyeReveal 직후 즉시 삭제
- DB에는 상태 추적용 최소 메타데이터만 남김
- 로그에도 민감 정보 기록 금지

효과:
- 서버 침해 시 유출되는 민감 정보 양 감소
- 장기 보관에 따른 프라이버시 리스크 완화

장점:
- 구현 난이도 낮음
- 게임 UX 영향 거의 없음

## 2. 전달 시점 늦추기

비공개 정보를 가능한 한 늦은 시점에 keeper에 전달하도록 설계한다.

예:
- 픽 정보는 commit 직후가 아니라 reveal 직전에 전달
- eye 순서는 eyeReveal 직전에만 전달

효과:
- keeper가 비공개 정보를 알고 있는 시간 자체를 줄임
- 유출/오남용 가능 시간 축소

장점:
- 게임 흐름 유지 가능
- 프라이버시 개선 대비 구현 복잡도 낮음

단점:
- 일부 단계에서 유저 액션 타이밍 관리가 조금 더 필요함

## 3. 암호화 저장

서버 DB에는 평문 대신 암호화된 형태로 저장하고, 실제 제출 직전에만 복호화한다.

예:
- `choices`, `order`, `salt`를 서버 측 암호화 키로 암호화 저장
- keeper 실행 시점에만 메모리 상에서 복호화 후 제출

효과:
- DB 유출 시 바로 평문이 노출되지 않음
- 기밀성 강화

장점:
- 서버 침해 리스크 완화에 효과적
- 게임 UX에는 영향 없음

단점:
- 키 관리 필요
- 구현 복잡도는 중간 수준

## 4. 키 분리

API 서버와 온체인 `operator` 서명 키를 분리한다.

예:
- 웹 API 서버는 사용자 요청만 수집
- 별도 worker 또는 isolated signer가 실제 `revealFor`, `eyeRevealFor` 트랜잭션 전송

효과:
- API 서버가 침해되어도 즉시 온체인 조작으로 이어질 가능성 감소
- 무결성 보호 강화

장점:
- 실질적인 보안 향상 대비 구현 부담이 크지 않음
- 운영 구조만 조금 분리하면 됨

## 5. 정보 저장 경로 분리

픽 정보와 eye 정보를 같은 저장소/프로세스에 두지 않고 분리한다.

예:
- `choices/salt` 저장 경로 분리
- `order/salt` 저장 경로 분리
- 가능하면 서로 다른 worker/process가 담당

효과:
- 한 번 침해로 모든 비공개 정보가 동시에 유출되는 위험 감소
- 피해 범위 제한

장점:
- 단일 실패 지점 완화
- 게임 플로우 변화 없음

단점:
- 운영 복잡도 다소 증가
- 배포 구조를 조금 더 나눠야 함

## 결론

현재 keeper 기반 구조는 재미와 UX를 유지하기 위한 현실적인 균형점에 가깝다.
따라서 다음 단계의 개선은 게임 플로우를 더 복잡하게 바꾸기보다, keeper의 저장 방식과 키 관리 방식을 강화하는 쪽이 더 적절하다.

추천 우선순위는 다음과 같다.

1. 저장 최소화
2. 전달 시점 늦추기
3. 키 분리
4. 암호화 저장
5. 정보 저장 경로 분리
